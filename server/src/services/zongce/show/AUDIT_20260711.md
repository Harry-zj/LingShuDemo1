## 一、当前真实完整流程

### 1.1 规则文件上传与解析

```
用户拖拽/选择 .docx 文件
  → SmartFillRule.onFiles()
    → api.uploadRuleFiles(formData)              POST /api/zongce/rules/upload
    → ruleController.uploadRuleFiles
      → INSERT rule_sources (status='pending')
  → emit('refresh') → SmartFill.refreshRules()
    → api.getRuleSources() + api.getRuleSets()

用户点击"🔍 解析"
  → SmartFillRule.doParse(sourceId)
    → api.parseRuleSource(sourceId)              POST /api/zongce/rules/sources/:id/parse
    → ruleController.parseRuleSource
      → INSERT ai_tasks (status='processing')
      → runParseV2InBackground(taskId, sourceId, userId)  // 后台异步
        → parseRuleSourceV2(sourceId, userId, onProgress)
          → extractStructure(filePath)           // mammoth HTML → doc_blocks
          → buildChapterTree(blocks)            // 章节识别（第X章）
          → buildParseTasks(blocks, chapters)   // 拆解析任务
          → splitLargeTasks(blocks, tasks)      // 大任务再拆分
          → 并行 chatStreamJson(DeepSeek)       // 每 task 调 AI
            Prompt: V2_RULE_PARSE_SYSTEM
          → mergeAndDedupResults()              // 合并去重
          → validateV2RuleParse()               // Schema 校验
          → validateMergedResult()              // 硬校验
          → 事务写入:
            INSERT indicator_nodes
            INSERT rule_packages
            INSERT executable_rules
            INSERT lookup_tables/dimensions/cells
            UPDATE rule_sources.status='parsed'
    → SSE /api/zongce/rules/tasks/:id/stream 推送进度
```

### 1.2 规则发布

```
用户点击"✅ 确认发布"
  → SmartFillRule.publishSet(rs)
    → api.publishRuleSet(rs.id)                  POST /api/zongce/rule-sets/:id/publish
    → ruleSetController.publishRuleSet
      → UPDATE rule_sets SET status='published', published_at=NOW()
  → 解锁材料/评分/填表步骤
```

### 1.3 材料上传与识别

```
用户创建材料 + 上传证明文件(.jpg/.png/.pdf)
  → SmartFillMaterial
    → api.createMaterial()                       POST /api/zongce/materials
    → api.uploadAttachments()                    POST /api/zongce/materials/:id/upload

用户点击"🔍 AI 识别"
  → SmartFillMaterial.doExtract(mat)
    → api.extractMaterial(mat.id)                POST /api/zongce/materials/:id/extract
    → materialController.extractMaterial
      → extractStructuredFacts(attachments)
        → 图片: ocrWithKimi(Kimi K2.6) → chatStreamJson(DeepSeek, MATERIAL_EXTRACT_SYSTEM)
        → 文档: 直接 chatStreamJson(DeepSeek, MATERIAL_EXTRACT_SYSTEM)
        → 返回 { facts: [...], needs_review, review_reason }
      → 持久化: INSERT/UPDATE material_recognitions.raw_ai_response (JSON)
      → UPDATE materials.title
    → 前端 mat._facts = res.data.facts
    → 自动 doPreview(mat, f) 每条 fact
```

### 1.4 规则匹配与分数预览

```
用户编辑 fact 字段 → @change 触发
  → SmartFillMaterial.doPreview(mat, fact)
    → api.previewScore(mat.id, { fact })         POST /api/zongce/materials/:id/preview
    → materialController.previewScore
      → matchFactPipeline(fact, ruleSetId)
        Phase 1: buildRuleBlockCards(ruleSetId)  // 生成规则卡片
        Phase 1: selectRuleBlocksWithKimi()      // DeepSeek 选 top-3
        Phase 2: loadRuleContext()               // 加载完整上下文
        Phase 3: matchFactWithKimi()             // DeepSeek 精确匹配
        → 返回 { selected_block, matched_rule_ids, score_dimensions, ... }
      → calculateScorePreview(fact, ruleSetId, pipelineMeta)
        → 优先用 matched_lookup_table_id 查表
        → 枚举归一化 + 维度名映射 → 生成 hash → 查 lookup_cells
        → 返回 { score, matched_rule, explanation, _debug }
    → 前端展示匹配结果 + 分数预览
```

### 1.5 确认入库

```
用户点击"✅ 确认并入库"
  → SmartFillMaterial.doConfirm(mat)
    → api.matchMaterial(mat.id, { fact })         POST /api/zongce/materials/:id/match
    → materialController.matchMaterial
      → UPDATE material_recognitions (category, explanation, confirm_status='pending')
    → 前端 mat.recognition = { ... }
```

### 1.6 评分计算（当前存在数据断链）

```
用户点击"计算分数"
  → SmartFillScore → emit('calculate')
  → SmartFill.calculateScore()
    → api.calculateScore()  ⚠ 无 body 参数
    → evaluationController.calculateScore
      → SELECT material_analysis_runs ⚠ 表永远为空，无代码写入
      → 返回 "没有可用的材料分析结果"
```

## 十二、删除计划

“当前没有调用或尚未实现”不等于“应该删除”。

必须结合当前真实调用链和 V2 目标流程，将相关代码、接口和数据库表分成以下五类：

1. 旧代码，但当前仍在运行；
2. 旧代码，已被新实现完整替代；
3. 新架构代码或表，但尚未接通；
4. 真正无用的代码；
5. 目前证据不足，无法确认。

重点区分：

- 无调用是因为已经被替代；
- 还是因为对应业务功能尚未实现；
- 还是仅缺少前端入口；
- 还是仅作为内部函数、兼容入口、恢复入口或后续计算阶段使用。
只有同时满足以下条件，才能标记为“可直接删除”：

- 不属于目标主流程；
- 当前无任何调用者；
- 没有兼容、恢复、迁移或测试用途；
- 已有完整替代实现；
- 替代流程已实际跑通；
- 删除后不会影响现有数据和接口。

如果只是推测，必须标记为“无法确认”，不能建议删除。

本轮不要修改代码，只更新审计报告和清理建议。
